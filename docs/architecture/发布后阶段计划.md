# 发布后阶段计划

本文档说明当前版本发布后的主线阶段，不再重复重构背景，而是回答“接下来先做什么、为什么先做、做到什么算完成”。

## 1. 当前判断

当前版本已经把浏览器动作主链从“能跑通”收口到“有基本产品边界”：

- 外接浏览器会话已区分“断开连接”和“关闭浏览器”
- HTTP 模式已明确为单会话串行
- 非幂等动作默认不自动重放
- 关键工具错误已开始结构化
- 基础 smoke 已覆盖主链与部分安全边界

这意味着项目已经不再适合继续只做架构重构，而应该进入“真实站点验收 -> 任务执行层 -> 产品化补完”的阶段推进。

## 2. 为什么按这三阶段推进

原因不是功能还不够多，而是当前风险已经从“代码能不能写出来”切换成下面三类：

1. 真实站点上是否还有未暴露的误判、误点、误收敛
2. 任务规格是否能真正跑成稳定执行链，而不只是样例 JSON
3. 发布、接入、错误口径、版本演进是否足够清晰

因此后续主线应按下面顺序推进：

1. 先做真实站点手动验收，暴露剩余问题
2. 再把任务执行层接起来，把经验沉淀成稳定流程
3. 最后做对外发布与长期维护层面的补完

## 3. 阶段总览

| 阶段 | 目标 | 输入 | 输出 | 完成标志 |
| --- | --- | --- | --- | --- |
| 第 1 阶段：真实站点手动验收 | 确认主链在公开网站上的稳定性 | 当前 MCP 能力、公开站点清单 | 真实站点验收记录、失败样例、修复清单 | 常见搜索/详情/新页/遮挡/媒体场景可稳定复现 |
| 第 2 阶段：任务执行层落地 | 把任务样例变成真正可执行流程 | `tests/web-tasks/*.json`、已有工具能力 | 任务执行器、结构化日志、失败报告 | 任务定义可直接执行并产出统一报告 |
| 第 3 阶段：产品化补完 | 收口发布、接入、维护能力 | 已稳定的 MCP 主链与任务执行层 | 版本发布流程、错误码文档、接入说明 | 项目具备可持续发布和对外说明能力 |

## 4. 第 1 阶段：真实站点手动验收

### 4.1 目标

目标不是“多跑几个网站”，而是把现有动作链在真实页面里按人工方式一条条压过去，找出仍然会误选、误判、误关闭、误摘要的点。

### 4.2 为什么先做这个

如果不先做这一阶段，后面直接做任务执行层，只会把当前尚未暴露的问题自动化放大。

### 4.3 具体流程

第 1 轮：

1. 选 8 到 12 个公开网站
2. 按场景分组覆盖：
   - 首页搜索
   - 结果页进入详情
   - 新页 / `popup` / 新 `target`
   - 同页刷新
   - 遮挡弹窗关闭
   - 视频播放
   - 正文提取
   - 结果卡片提取
3. 全程使用可见浏览器，按真实人工动作逐步执行
4. 每轮记录：
   - 命中的真实元素
   - 动作后发生变化的真实页面
   - 成功信号来自哪里
   - 如果失败，失败发生在哪一层

第 2 轮：

1. 针对第 1 轮暴露的问题改代码
2. 只复测失败站点和相邻场景
3. 把结论沉淀到测试清单、夹具或规则文档

### 4.4 关键输入输出

输入：

- 当前工具集与默认配置
- 公开站点验收清单
- 人工可见浏览器会话

输出：

- 站点级验收记录
- 误判模式清单
- 修复优先级
- 可沉淀为规则的结论

### 4.5 状态变化

从：

- “本地 smoke 通过”

变成：

- “真实站点主链可稳定复现”

### 4.6 边界和不变式

- 不用脚本替代人工验证主线
- 不以历史 selector 记忆替代当前页面事实
- 输入框已确认后，只切换提交策略，不回头怀疑输入框层
- 结果提取必须绑定到真正发生变化的页面

### 4.7 第一版先做什么

- 完成一版 10 个左右真实站点的人工验收
- 补齐失败记录和对应修复项

### 4.8 后续增强

- 将高价值站点沉淀成长期回归任务
- 建立场景覆盖矩阵与回归频次

## 5. 第 2 阶段：任务执行层落地

### 5.1 目标

把 `tests/web-tasks/*.json` 从“任务样例”升级为“可以直接执行的任务定义”。

### 5.2 为什么放在第 2 阶段

因为任务执行层的价值不是再造一套动作，而是把第 1 阶段已经确认过的动作链和判定逻辑装配起来。没有前一阶段的事实支撑，任务执行层会很脆。

### 5.3 具体流程

第 1 轮：

1. 定义统一任务结构：
   - 任务元信息
   - 步骤列表
   - 每步工具调用
   - 成功判据
   - 失败时保留的证据
2. 做最小执行器：
   - 读取任务
   - 顺序执行步骤
   - 记录每步结果
   - 失败即停并输出结构化报告

第 2 轮：

1. 增加任务级摘要
2. 增加截图、正文、结果卡片等产物归档
3. 增加可复用的步骤片段或模板

### 5.4 关键输入输出

输入：

- 任务规格文档
- 任务样例 JSON
- 已稳定的 MCP 工具调用结果

输出：

- 任务执行器
- 结构化执行日志
- 失败报告
- 任务产物目录

### 5.5 状态变化

从：

- “手工可以跑”

变成：

- “任务定义可以稳定执行并产出报告”

### 5.6 边界和不变式

- 执行器只编排，不重新发明页面理解逻辑
- 成功判据优先复用 MCP 主链已有结果
- 任务失败必须留下足够证据，不能只返回一句失败

### 5.7 第一版先做什么

- 打通 3 到 5 个典型任务样例
- 跑通读取、执行、记录、失败报告四件事

### 5.8 后续增强

- 增加任务模板库
- 增加批量任务与汇总报告

## 6. 第 3 阶段：产品化补完

### 6.1 目标

把“能开发和验证”推进到“能持续发布、能稳定接入、能清楚说明边界”。

### 6.2 为什么放在最后

这一阶段依赖前两阶段已经把真实行为和执行口径收稳；否则对外文档、错误码和发布说明很快会失真。

### 6.3 具体流程

第 1 轮：

1. 补全版本与发布链路：
   - Git tag
   - GitHub Release
   - 变更日志
2. 补全文档：
   - 接入说明
   - 错误码说明
   - 已知限制
   - 典型使用示例

第 2 轮：

1. 评估是否需要 npm 发布
2. 评估 HTTP 模式是否从“单会话串行”升级到“多会话”
3. 补更稳定的 CI 与回归矩阵

### 6.4 关键输入输出

输入：

- 稳定版本代码
- 变更日志
- 测试与验收结论

输出：

- 发布标签与 Release
- 对外文档
- 错误码与边界说明
- 后续版本节奏

### 6.5 状态变化

从：

- “工程化 Beta”

变成：

- “具备明确发布与维护口径的产品版本”

### 6.6 边界和不变式

- 发布说明必须基于当前已实现能力，不夸大
- 错误码文档必须与真实返回结构一致
- 对外能力说明必须明确区分“已稳定支持”和“仍在演进”

### 6.7 第一版先做什么

- 完成当前版本的 tag 与 Release
- 补阶段计划、版本说明、错误码和已知限制文档

### 6.8 后续增强

- npm 发布
- 多会话 HTTP 设计
- 更完整的 CI / 回归基线

## 7. 建议执行顺序

当前建议只按下面顺序推进，不要跳步：

1. 先完成真实站点手动验收
2. 再把任务执行层做成可运行能力
3. 最后集中补发布与产品化说明

## 8. 当前阶段结论

对当前版本而言，当前最重要的不是继续大拆架构，而是：

1. 用真实站点把主链剩余问题压出来
2. 把任务定义真正接成执行链
3. 再把发布和接入口径补完整

这三阶段完成后，项目才会从“主链已经成形”进一步进入“真实可交付、可持续演进”的状态。
