## 2. 实施目标

第一阶段不是把所有能力推倒重来，而是先把最容易导致实站失败的 3 条底层主线拆开：

1. 元素身份与目标绑定
2. 动作执行与命中证明
3. 结果观察与最终判定

只要这 3 条主线收住，后续 `submitPlan / dismissPlan / openResultPlan / playMediaPlan` 才有稳定底座。

结合 `browser-use` 源码调研，当前实施顺序再补一条前置约束：

- 在真正重做“绑定、执行、观察”前，要先把统一状态快照和 `target/session` 事实源补起来；
- 否则上层 3 条主线仍然会建立在零散 DOM 读取和单一 `page` 句柄假设上。

## 3. 分阶段实施

## 3.0 阶段零：补统一状态快照与会话事实层

目标：

1. 统一产出 DOM、AX、必要样式、几何与交互证据。
2. 在会话层明确区分 `session / target / page`。
3. 为后续绑定层和观察层提供统一事实源。

主要改动文件：

1. [session.ts](/D:/C_Project/chrome_mcp_server/src/browser/core/session.ts)
2. 新增 `src/browser/state/*`
3. 新增 `src/browser/session/*`

阶段完成标准：

1. `page_snapshot`、`find_elements`、`click_and_wait` 不再各自零散抓取页面状态。
2. `popup / new_target / target_switched` 成为统一状态事件，而不是各工具各自猜测。
3. discovery、binding、observation 三层都消费同一份状态快照结构。

## 3.1 阶段一：重做目标绑定

目标：

1. 停止让 `ref` 直接等于 selector。
2. 引入稳定元素身份：
   - `runtimeNodeKey`
   - `fingerprint`
   - `contextAnchor`
   - `fallbackAnchors`
3. 动作前支持按绑定记录重新定位真实 DOM。

主要改动文件：

1. [session.ts](/D:/C_Project/chrome_mcp_server/src/browser/core/session.ts)
2. [dom-helpers.ts](/D:/C_Project/chrome_mcp_server/src/browser/core/dom-helpers.ts)
3. 新增 `src/browser/binding/*`

阶段完成标准：

1. `find_elements` 返回的不再只是“可点 selector”，而是带运行时节点身份和语义锚点的候选。
2. `ref` 不再因为重复 selector 串位。
3. 页面未大变时可优先按 `runtimeNodeKey` 命中目标；失效后再回退到语义绑定。
4. GitHub 文件列表、搜索结果卡片、表格行等重复结构能稳定区分目标。

## 3.2 阶段二：重做统一动作执行器

目标：

1. 点击、输入、按键统一走同一条执行链。
2. 动作执行前必须有 preflight。
3. 动作执行后必须产出命中证据。
4. 语义点击失败时允许降级到受约束的安全坐标点击。

主要改动文件：

1. [interaction-tools.ts](/D:/C_Project/chrome_mcp_server/src/browser/ops/interaction-tools.ts)
2. 新增 `src/browser/execution/*`
3. 新增 `src/browser/usecases/*`

执行器至少要产出这些信息：

1. 是否真正发出了动作
2. 命中点坐标
3. 顶层命中的真实元素
4. 是否被遮挡
5. 是否命中目标
6. 是否走了坐标降级

阶段完成标准：

1. “没点中”和“点中了但没跳”能明确区分。
2. 点击失败时能直接知道是遮挡、命中偏移、目标不存在还是执行异常。
3. `click / type_text / press_key` 都能复用同一套执行骨架。
4. 对已明确绑定但语义点击不稳的站点，能在安全约束下回退到坐标点击，而不是直接判失败。

## 3.3 阶段三：拆分观察层与判定层

目标：

1. 观察层只采集证据，不判成功失败。
2. 判定层只根据意图和证据下结论。
3. 重试策略不再藏在观察逻辑内部。
4. 观察层显式依赖 `target/session` 事件，而不是只盯当前 `page`。

主要改动文件：

1. [action-execution-usecases.ts](/D:/C_Project/chrome_mcp_server/src/browser/usecases/action-execution-usecases.ts)
2. [action-observation.ts](/D:/C_Project/chrome_mcp_server/src/browser/observation/action-observation.ts)
3. [action-judgement.ts](/D:/C_Project/chrome_mcp_server/src/browser/judgement/action-judgement.ts)

阶段完成标准：

1. 页面变化已经发生时，不会再因为判定收敛慢而回头重试旧目标。
2. `same_page_update / navigation / popup / new_target` 能统一归因。
3. `overlay / verification / auth` 能和普通失败分开。
4. 活动 `target` 切换、目标页新开、原页未变这几类场景能直接落到证据层。

## 3.4 阶段四：接回现有 plan

目标：

1. 让 `submitPlan / dismissPlan / openResultPlan / playMediaPlan` 消费稳定目标，而不是直接消费脆弱 selector。
2. 让现有高级工具继续保留，但底层换成新骨架。

主要改动文件：

1. [submit-usecases.ts](/D:/C_Project/chrome_mcp_server/src/browser/usecases/submit-usecases.ts)
2. [overlay-usecases.ts](/D:/C_Project/chrome_mcp_server/src/browser/usecases/overlay-usecases.ts)
3. [find-primary-results.ts](/D:/C_Project/chrome_mcp_server/src/browser/discovery/find-primary-results.ts)
4. [media-state.ts](/D:/C_Project/chrome_mcp_server/src/browser/discovery/media-state.ts)

阶段完成标准：

1. plan 的顺序能力保留。
2. plan 的目标定位切到新绑定体系。
3. plan 的成功判定切到新观察/判定体系。

## 4. 迁移顺序建议

建议按这个顺序推进，不要并行乱拆：

1. 先补统一状态快照与 `target/session` 事实层。
2. 再引入新的绑定类型与绑定注册表，但保留旧 selector 兜底。
3. 再把 `click` 和 `click_and_wait` 切到新执行器。
4. 再切 `type_text`、`press_key_and_wait`。
5. 再把动作验证编排入口收口到 `usecases/action-execution-usecases.ts`。
6. 最后把上层 plan 接进新骨架。

原因：

1. `click` 是当前问题最集中的入口。
2. 只要 `click` 还没收住，后面测 plan 只会持续被噪音干扰。
3. 但 `click` 要收住，前提是先统一状态事实源；否则绑定层和观察层仍会各自读一套页面现实。

## 5. 需要保留的兼容策略

第一阶段应保留这些兼容策略，避免大面积破坏现有工具：

1. 外部 MCP 工具名先不改。
2. 调用参数里仍允许传 `selector`。
3. 旧 selector 仍作为 fallback，不立刻删除。
4. 现有 smoke 和夹具先尽量复用，再逐步迁移到新返回结构。

## 6. 每阶段验收方式

## 6.1 阶段一验收

至少验证这些场景：

1. 重复文件列表中的唯一文件点击
2. 搜索结果列表中的标题链接绑定
3. 弹窗右上角关闭按钮与 CTA 区分
4. `label` 包裹输入框、`span` 包裹控件这类代理交互节点能正确绑定

## 6.2 阶段二验收

至少验证这些场景：

1. 目标被遮挡时，点击前就能识别
2. 目标不在视口时，滚动后能再次命中预检
3. 点击后能拿到顶层命中元素与点击点信息
4. 语义点击失败但目标明确时，能安全降级到坐标点击

## 6.3 阶段三验收

至少验证这些场景：

1. 页面已经跳转，但内容稍后就绪
2. 同页局部刷新而不是 URL 变化
3. popup / new target 打开后返回真实变化页
4. 活动 target 切换但原 page 未变化时仍能正确归因

## 6.4 阶段四验收

至少验证这些场景：

1. `submitPlan` 不再因为旧 selector 漂移导致重复怀疑输入框
2. `dismissPlan` 先区分真实关闭目标与高风险 CTA
3. `openResultPlan` 优先命中标题链接而不是泛化缩略图
4. `playMediaPlan` 能在详情页和媒体状态之间形成闭环
