# 前端基础支持学习笔记

## 1. 文档定位

这份文档不是重构方案，也不是实施清单。

它的目的只有一个：

- 把当前 `chrome_mcp_server` 真正需要补的前端 `HTML / CSS / DOM / 可访问性` 基础能力先学明白、记下来，后续改代码时不再凭感觉猜。

本文重点不是“前端教程”，而是：

1. 哪些基础知识与我们当前项目直接相关
2. 这些知识会影响哪一层实现
3. 后续应该把哪些基础能力补进底层

## 2. 这轮补了哪些基础

这轮优先补的是和“选目标、点目标、判是否被遮挡、判是否是表单提交、判弹窗是否可关闭”直接相关的内容：

1. `HTML` 语义元素
   - `<a>`
   - `<button>`
   - `<label>`
   - `<input>`
   - `<form>`
   - `<dialog>`

2. DOM 命中测试
   - `document.elementFromPoint()`
   - `document.elementsFromPoint()`

3. `CSS` 可点击性与遮挡判断
   - `pointer-events`
   - `visibility`
   - `opacity`
   - `overflow`
   - `stacking context`
   - `z-index`

4. 可访问性命名
   - `accessible name`
   - `aria-label`
   - `aria-labelledby`
   - `label -> control` 关联

## 3. 学到的关键结论

## 3.1 `<a>` 只有带 `href` 时才是真正的链接

对我们最重要的点：

1. `<a>` 带 `href` 时才有真正的超链接语义。
2. 这时它的默认可访问角色是 `link`。
3. 聚焦后按 `Enter` 会激活。

对项目的影响：

1. 选结果页标题链接时，要优先认 `a[href]`。
2. 没有 `href` 的 `<a>`，不能默认当成可导航目标。
3. 目标绑定时，`href` 应是最强锚点之一。

## 3.2 `<button>` 在表单里默认可能就是提交按钮

对我们最重要的点：

1. `<button>` 如果位于或关联到 `<form>`，且没有 `type="button"`，它可能就是提交按钮。
2. 不能看到按钮文字不是“搜索/提交”就直接排除它的提交语义。

对项目的影响：

1. `find_submit_targets` 不能只看按钮文本。
2. 还要看它是否处在 form 里，是否为默认 submit 行为。
3. `submitPlan` 需要把“form 关联”和“button type”纳入判断。

## 3.3 `<label>` 不只是文本，它会把焦点和激活转交给控件

对我们最重要的点：

1. `<label for="x">` 会和 `id="x"` 的第一个可 label 控件建立关联。
2. 也可以通过“把控件包在 `<label>` 里”形成隐式关联。
3. 用户点击 `label` 时，浏览器会把焦点/激活转交给关联控件。
4. 如果点击的是 `label` 里的其他交互元素，`label` 的默认激活行为应不起作用。

对项目的影响：

1. 输入框识别不能只盯 `<input>` 自己。
2. 必须补“显式 label 关联”和“隐式 label 包裹”两种解析。
3. 目标绑定时，`label` 文本是搜索框、登录框、筛选项等控件的重要锚点。

## 3.4 可访问名称不是“有没有 aria-label”这么简单

对我们最重要的点：

1. 控件需要可访问名称。
2. 名称可能来自：
   - 元素自身文本
   - `<label>`
   - `aria-labelledby`
   - `aria-label`
   - 某些元素的专用属性，如 `alt`、`title`
3. 若 `aria-labelledby` 和 `aria-label` 同时存在，`aria-labelledby` 优先。
4. `aria-label` 适合“没有可见标签文本”时使用，不应被过度滥用。

对项目的影响：

1. 我们现在的匹配分数过于粗糙。
2. 后面要单独实现“可访问名称解析器”，不能只把 `aria-label` 当普通属性。
3. 目标绑定的优先级应从“可访问名称”出发，而不是从 `class` 出发。

## 3.5 `pointer-events:none` 的元素本身不应被视为点击目标

对我们最重要的点：

1. `pointer-events: none` 会让事件“穿透”当前元素，落到它下面的元素。
2. `elementFromPoint()` 会忽略这类元素，直接返回下面的元素。
3. 但父节点仍可能在事件捕获/冒泡链上收到事件。

对项目的影响：

1. 遮挡层判断时，不能只看“视觉上盖住了页面”。
2. 如果一个层是 `pointer-events:none`，它不一定真的拦住了操作。
3. 这也是为什么之前某些 banner 被误判成 `overlay_blocking` 是错的。

## 3.6 `elementFromPoint()` / `elementsFromPoint()` 是点击命中的基础

对我们最重要的点：

1. `elementFromPoint(x, y)` 返回视口坐标处最上层的命中元素。
2. `elementsFromPoint(x, y)` 返回该点从上到下的所有元素。
3. 它们是判断“这一下点到了谁”“上面是不是还有遮挡层”的基础工具。

对项目的影响：

1. 点击前必须做 hit-test，而不是直接 `locator.click()`。
2. 点击后也要保留命中证据：
   - 点击点
   - 顶层元素
   - 目标元素是否出现在 `elementsFromPoint()` 链里
3. 这会直接决定“点中了没”和“被谁挡住了”的归因质量。

## 3.7 `stacking context` 说明 `z-index` 不是全局数字大小比较

对我们最重要的点：

1. `z-index` 只在当前 stacking context 内有意义。
2. 一个子 stacking context 在父层里是整体参与叠放的。
3. 所以“哪个元素更高”不能只比两个数字。

对项目的影响：

1. 遮挡层和顶部命中判断，不能用简单的 `z-index` 数值大小硬比。
2. 真正可靠的仍然应该是 `elementFromPoint()`。
3. `z-index` 只能作为辅助信号，而不是最终真理。

## 3.8 `overflow` 会影响是否可见、是否可点击、是否需要滚动

对我们最重要的点：

1. `overflow: clip / hidden` 会裁掉内容。
2. `overflow: auto / scroll` 会形成滚动容器。
3. 元素即使存在于 DOM 里，也可能因为被父容器裁掉而不可见或不可直接点击。

对项目的影响：

1. 可见性判断不能只看元素自己的 `rect`。
2. 还要看它是不是被祖先滚动容器裁掉了。
3. 后续滚动逻辑应该优先滚最近的滚动容器，而不是默认整页滚动。

## 3.9 `<dialog>` 和普通 `div modal` 不完全一样

对我们最重要的点：

1. `<dialog>` 有原生语义。
2. `showModal()` 会把它提升为模态层。
3. `::backdrop` 是它的原生背景遮罩。
4. 某些场景支持点击外部关闭或 `Esc` 关闭，但是否允许这样做，和具体行为有关。

对项目的影响：

1. 遮挡层分类时，原生 `<dialog>` 应优先单独识别。
2. `backdrop_click` 不应再完全靠页面 heuristics 猜。
3. 如果页面就是原生 `dialog`，我们应优先判断它的模态行为，而不是把它当普通 `div` 弹窗。

## 4. 对当前项目最直接的补强方向

## 4.1 先补“语义优先”的候选构建

后续 discovery 层应该优先提取这些锚点：

1. 元素语义：
   - `a[href]`
   - `button`
   - `input`
   - `textarea`
   - `select`
   - `label`
   - `dialog`

2. 稳定属性：
   - `id`
   - `name`
   - `for`
   - `title`
   - `href`
   - `type`
   - `placeholder`
   - `role`
   - `aria-label`
   - `aria-labelledby`

3. 上下文信息：
   - 所属 `form`
   - 所属 `tr`
   - 所属卡片容器
   - 所属 `dialog/modal`

`class` 和长路径 selector 应退到兜底层，不再做主锚点。

## 4.2 补一个真正的 label / accessible-name 解析器

这是当前项目最缺的基础之一。

至少要支持：

1. `label[for=id]`
2. `<label><input ... /></label>` 隐式关联
3. `aria-labelledby`
4. `aria-label`
5. 元素自身文本

这样输入框、筛选框、登录框、开关项的识别才会稳。

## 4.3 补一个真正的 hit-test 工具

执行层至少要统一提供：

1. 计算点击点
2. `elementFromPoint`
3. `elementsFromPoint`
4. 判断目标是否在命中链上
5. 判断上层拦截元素属于：
   - 遮挡层
   - 普通容器
   - CTA
   - 弹窗背景

这会是“点击前预检”和“点击后证据”的共同底座。

## 4.4 补“滚动容器感知”

后续不能默认所有目标都靠整页滚动解决。

至少要补：

1. 识别最近滚动容器
2. 判断元素是否被祖先容器裁切
3. 优先滚容器，再滚页面

## 4.5 补“原生弹窗语义识别”

后续遮挡层逻辑里，至少要单独识别：

1. `<dialog>`
2. 原生表单关联控件
3. `label` 触发的控件焦点切换
4. `button` 的 submit 行为

这样很多“看起来像自定义 div”的地方，才能先按 HTML 原生规则理解，而不是直接靠 heuristics 猜。

## 6. 当前结论

这轮“前端基础支持”的学习结论是：

1. 当前项目最缺的不是更多 heuristics，而是更扎实的前端语义基础。
2. 后续真正要补的是：
   - 语义化候选提取
   - label / accessible name 解析
   - hit-test
   - stacking / overflow / pointer-events 感知
   - 原生弹窗与表单行为识别
3. 这些基础补完后，再去做目标绑定和动作执行层，代码才不会继续建立在脆弱猜测上。
