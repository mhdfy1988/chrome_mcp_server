## 1. 背景

当前 `chrome_mcp_server` 已经具备一套可工作的浏览器自动化主路：

1. 页面快照与元素查找
2. 点击、输入、提交、截图、提取
3. 动作后的页面变化观察
4. 遮挡层、结果页、媒体播放等专项计划

这套主路在一部分站点上已经能跑通，但最近几轮真实站点实测也暴露出一个更根本的问题：

- 问题不再只是某个 heuristic 不够，而是底层架构把多种职责揉在了一起
- 同一个失败现象，可能来自元素定位、动作执行、页面观察、成功判定中的任意一层
- 当前实现无法稳定回答这三个最关键的问题：
  - 我们到底锁定了哪个元素
  - 这一下动作到底有没有真正打到它
  - 页面后续变化到底说明了什么

因此，后续不应继续沿现有主线堆规则，而应把浏览器动作链拆层重构。

## 2. 目标

本次重构的目标是：

1. 把“元素发现、目标绑定、动作执行、结果观察、最终判定”拆成清晰的 5 层。
2. 让 `ref` 不再等价于 selector，而是成为可重新绑定的元素身份。
3. 让点击、输入、按键等动作都具备“动作是否真正落到目标”的证据链。
4. 让动作后的“页面变化”和“最终是否成功”分开建模。
5. 让上层 `submitPlan / dismissPlan / openResultPlan / playMediaPlan` 建立在稳定底座上，而不是继续依赖脆弱 selector。

## 3. 非目标

第一版重构不追求：

1. 一次性解决 iframe、shadow DOM、虚拟列表、跨域嵌套等全部复杂场景。
2. 跨页面、跨导航、跨浏览器重启永久保持同一个 `ref`。
3. 立刻完全废弃 selector。
4. 一次性重写全部工具对外接口。

第一版只追求：

1. 让主路中的点击、输入、提交三类动作先变稳。
2. 让页面变化观察与成功判定先拆开。
3. 让真实站点中的常见失败能明确归因到具体一层。

## 4. 当前痛点

## 4.1 元素身份不稳定

当前 `ref` 生成与存储逻辑位于 [session.ts](/D:/C_Project/chrome_mcp_server/src/browser/core/session.ts)。

现状问题：

1. `ref` 本质上仍依赖 selector。
2. 同一个 selector 会被复用成同一个 `ref`。
3. 当 selector 不唯一时，不同行、不同卡片、不同文件链接会共用一个 `ref`。

真实表现：

- GitHub 文件列表中，`README.md`、`authors.yaml`、`registry.yaml` 等文件链接都可能退化到同一条公用 selector。
- 页面快照看起来“找到了 authors.yaml”，但点击时实际并不能唯一落到它。

结论：

- 当前 `ref` 不是“元素身份”，只是“selector 缓存别名”。

## 4.2 元素发现与动作执行耦合

当前元素发现主逻辑位于 [find-elements.ts](/D:/C_Project/chrome_mcp_server/src/browser/inspect/find-elements.ts)。

现状问题：

1. `accessibleName / label / text / placeholder / title / class / href / selector` 全部混成一个匹配分数。
2. 这套机制适合“给候选”，不适合“直接拿来执行动作”。
3. 上层经常把“模糊命中的候选元素”直接送给点击或输入。

真实表现：

- 能找到“像目标”的元素，但不能保证它就是当前操作该打的那个目标。
- 一旦页面里存在重复结构、重复 class、重复图标，这条链很容易漂移。

结论：

- 现在缺少一层真正的“目标绑定层”。

## 4.3 selector 生成策略天然脆弱

当前 selector 生成逻辑位于 [dom-helpers.ts](/D:/C_Project/chrome_mcp_server/src/browser/core/dom-helpers.ts)。

即使已经做过一轮增强，整体问题仍然存在：

1. 核心思路仍是“尽量生成一条看起来可用的 selector”。
2. 页面一旦重排、局部刷新、重复节点增减，selector 稳定性就下降。
3. selector 适合兜底，不适合承担“元素身份”职责。

结论：

- selector 应降级为“绑定与执行的辅助信息”，而不是核心主键。

## 4.4 点击执行缺少命中证明

当前点击主逻辑位于 [interaction-tools.ts](/D:/C_Project/chrome_mcp_server/src/browser/ops/interaction-tools.ts)。

现状问题：

1. 现在是拿到 selector 或 ref 后直接调用 `locator.click()`。
2. 缺少点击前预检：
   - 目标是否唯一
   - 目标是否可见
   - 目标中心点上方是否真的是它
   - 是否被遮挡
3. 缺少点击后回执：
   - 是否实际执行
   - 命中点坐标是多少
   - 顶层命中的真实节点是谁

真实表现：

- 看起来“执行了点击”，但实际上并不能证明点中目标。
- 页面没变时，不知道是没点中、站点无响应，还是后续判定没识别出来。

结论：

- 当前点击执行层没有“动作已命中”的证据链。

## 4.5 观察、判定、重试揉成一团

当前动作后的统一编排入口位于 [action-execution-usecases.ts](/D:/C_Project/chrome_mcp_server/src/browser/usecases/action-execution-usecases.ts)，其内部再调用 [action-observation.ts](/D:/C_Project/chrome_mcp_server/src/browser/observation/action-observation.ts) 与 [action-judgement.ts](/D:/C_Project/chrome_mcp_server/src/browser/judgement/action-judgement.ts)。

现状问题：

1. 动作执行、页面观察、成功判定、失败原因、重试策略都在一条链里。
2. “动作有没有打到目标”和“页面有没有发生变化”没有被分开建模。
3. “页面变化了但没判成功”时，可能继续拿旧目标重试。

真实表现：

- `Wikipedia` 搜索页中，动作已成功跳到词条页，但观察层未及时认定成功。
- 工具后续仍可能继续沿旧目标或旧条件等待。

结论：

- 当前观察层承担了过多职责，必须拆分。

## 4.6 页面状态分类会反向污染动作链

当前页面状态判断位于 [page-state.ts](/D:/C_Project/chrome_mcp_server/src/browser/core/page-state.ts) 及相关遮挡层逻辑。

现状问题：

1. `overlay_blocking / auth_required / blocked_by_verification / normal` 这些状态会直接影响后续动作。
2. 一旦页面状态误判，后续动作、等待与关闭策略都会被整体带偏。

真实表现：

- GitHub 顶栏搜索入口曾被误判成 `overlay_blocking`。
- B 站首页普通层和登录拦截层容易互相污染。

结论：

- 页面状态分类应该变成观察证据的一部分，而不是直接统治动作执行。

## 4.7 上层 plan 已存在，但底座还不稳

当前已经有：

1. `submitPlan`
2. `dismissPlan`
3. `openResultPlan`
4. `playMediaPlan`

问题在于：

1. 这些 plan 仍然消费脆弱的 selector/ref。
2. plan 决策本身没问题，但底层目标绑定和动作执行不稳，就会把 plan 带偏。

结论：

- plan 不是问题本身，问题在于 plan 的底座还没拆干净。

## 5. 根因总结

一句话总结当前架构问题：

当前实现没有把下面这三件事分开：

1. 元素是谁
2. 动作有没有真的打到它
3. 页面后来发生了什么

只要这三件事继续混在一起，后面继续补规则，仍然会在真实站点上反复暴露同类问题。

## 6. 重构后的目标架构

建议重构为 5 层：

1. 页面发现层（Discovery）
2. 目标绑定层（Binding）
3. 动作执行层（Execution）
4. 结果观察层（Observation）
5. 结果判定层（Judgement）

这 5 层的关系是：

```text
页面 -> 候选元素 -> 已绑定目标 -> 动作尝试 -> 观察证据 -> 最终判定
```

## 6.1 结合 browser-use 源码后的修订

在对照 `browser-use` 的 `mcp/server.py`、`browser/session.py`、`dom/service.py`、
`dom/serializer/clickable_elements.py`、`browser/profile.py` 后，当前方案建议做 5 处修订。

注意：

1. 主体仍然保留“发现、绑定、执行、观察、判定”这 5 层。
2. 不建议照搬 `state + index` 的对外交互形式。
3. 需要借的是它的状态建模、会话建模和 DOM/AX/CDP 组合方式。

### 修订一：补一个统一状态快照基础层

当前方案默认“页面发现层”直接面向页面取数，这还不够稳。

参考 `browser-use` 在 `dom/service.py` 里并行采集：

1. DOM tree
2. DOM snapshot
3. AX tree
4. 必要 computed styles
5. paint order
6. JS click listener 证据

建议我们在 5 层业务架构下面，再补一个共享基础层：

- 统一状态快照层（State Snapshot Foundation）

职责：

1. 统一产出页面状态基底，而不是让 `page_snapshot`、`find_elements`、`click`、`dismiss` 各自零散读 DOM。
2. 给上层提供结构化节点、可访问性语义、几何信息、样式信息和交互证据。
3. 作为页面发现层、目标绑定层、观察层共同依赖的单一事实来源。

这样调整后，5 层的关系变成：

```text
统一状态快照基础层
  -> 页面发现
  -> 目标绑定
  -> 动作执行
  -> 结果观察
  -> 结果判定
```

### 修订二：目标绑定不能只靠 fingerprint，要补运行时节点身份

当前方案里 `BoundTarget` 主要强调：

1. `fingerprint`
2. `contextAnchor`
3. `fallbackAnchors`

这条方向是对的，但还不够。

参考 `browser-use` 在 `dom/service.py` 里通过 CDP 把对象解析到 `backendNodeId` 的做法，
我们应把目标身份拆成两层：

1. 运行时节点身份
   - 例如 `backendNodeId`
   - 或目标页内可重新解析的 `runtimeNodeKey`
2. 稳定语义身份
   - `fingerprint`
   - `contextAnchor`
   - `fallbackAnchors`

结论：

1. `fingerprint` 负责“这是谁”。
2. `runtimeNodeKey` 负责“当前这一轮执行要打哪一个真实节点”。
3. 页面未大变时优先复用运行时节点身份；失效后再回退到语义绑定。

### 修订三：页面发现层要引入“交互语义检测”，不只是文本匹配

参考 `clickable_elements.py`，它不仅看标签和文本，还会看：

1. JS click listener
2. label 包裹控件
3. span 包裹表单控件
4. ARIA role / AX role
5. cursor pointer
6. 搜索类 class / id / data-* 信号

这说明我们当前文档里“页面发现层”的字段还偏弱。

建议把发现层的输入输出加强为：

1. 不只产出文本和属性命中，还要产出交互证据：
   - `hasJsClickListener`
   - `axRole`
   - `cursorStyle`
   - `isLabelProxy`
   - `formControlRelation`
2. 发现层不要直接打总分，而应先做“语义归类”，再做候选排序。
3. 搜索框、提交按钮、关闭按钮、媒体按钮等专项能力，优先建立在交互语义上，而不是裸文本。

### 修订四：观察层要显式依赖 target/session 事件，不只盯 page 变化

参考 `browser/session.py`，`browser-use` 明确区分：

1. browser session
2. target
3. tab / page

并且是事件驱动的。

这对我们当前方案的直接修订是：

1. 观察层不应只盯“当前 page 有没有变”。
2. 应显式采集这些证据：
   - 新 target
   - target 激活切换
   - tab 创建/关闭
   - navigation started / completed
3. 页面句柄只是结果载体，不应再充当唯一事实源。

也就是说，我们现在文档里“结果观察层”还需要再明确：

- `page`、`target`、`session` 是三层不同对象
- `popup/new_target/active target switched` 应成为一等证据

### 修订五：执行层要内建“语义点击 -> 坐标点击”的降级梯度

参考 `browser-use` 的本地 MCP，点击既支持按索引，也支持按坐标。

我们不需要照搬 index，但这个设计说明一件事：

- 执行层必须同时支持“目标语义点击”和“像素级命中点击”

所以当前执行层方案建议补成下面这条梯度：

1. 优先按绑定目标执行语义点击
2. 语义点击前做 hit-test 预检
3. 若目标已确认但语义点击不稳定，可降级到安全坐标点击
4. 坐标点击后仍要回收 `topElementAtPoint`

这不是鼓励“盲目坐标点击”，而是把它变成有约束的最后一跳。

## 6.2 对当前主方案的直接影响

结合上面 5 处修订，当前主方案需要做这 4 个直接调整：

1. 保留 5 层业务架构，但补一个共享的“统一状态快照基础层”。
2. `BoundTarget` 要从“纯语义绑定”升级成“运行时节点身份 + 语义绑定”的双层结构。
3. 页面发现层要从“文本/属性打分”升级成“交互语义归类 + 候选排序”。
4. 结果观察层和会话层之间要增加明确的 `target/session` 事件接口，而不是只通过 `page` 句柄间接推断。

## 7. 五层职责设计

## 7.1 页面发现层

目标：

- 只负责回答“页面上有哪些候选交互目标”

输入：

- 当前页面
- 查询词
- 可选 role/tag 约束

输出：

- `ElementCandidate[]`

建议字段：

1. `semanticRole`
2. `text`
3. `accessibleName`
4. `title`
5. `ariaLabel`
6. `href`
7. `name`
8. `placeholder`
9. `context`
10. `rect`
11. `visibility`
12. `riskTags`
13. `axRole`
14. `hasJsClickListener`
15. `cursorStyle`
16. `formControlRelation`

注意：

- 这一层只产出候选，不负责决定“到底点哪个”。

## 7.2 目标绑定层

目标：

- 把“候选元素”绑定成“本次动作的真实目标”

输入：

- `ElementCandidate`
- 当前页面快照
- 动作意图

输出：

- `BoundTarget`

建议字段：

1. `targetId`
2. `runtimeNodeKey`
3. `fingerprint`
4. `primaryAnchor`
5. `contextAnchor`
6. `fallbackAnchors`
7. `expectedRole`
8. `expectedText`
9. `expectedHref`
10. `rect`

关键原则：

1. `ref` 不再直接等于 selector。
2. `fingerprint` 应由多种稳定锚点共同组成，例如：
   - `title`
   - `aria-label`
   - `href`
   - 明确文本
   - 所属行或卡片容器
3. 每次执行动作前，都要用 `fingerprint + contextAnchor` 重新绑定一次真实 DOM。

## 7.3 动作执行层

目标：

- 像人一样把动作真正落到目标上

输入：

- `BoundTarget`
- 动作类型

输出：

- `ActionAttempt`

建议字段：

1. `executed`
2. `hitTarget`
3. `clickedPoint`
4. `topElementAtPoint`
5. `blockedBy`
6. `usedStrategy`
7. `attemptedSelector`
8. `attemptedRuntimeNodeKey`
9. `note`

执行前必须做的预检：

1. 目标是否仍存在
2. 是否唯一
3. 是否可见
4. 是否在视口中
5. 命中点顶部元素是不是目标或其后代
6. 是否被遮挡层挡住

这层的结论不是“任务成功了”，而是：

- 这一下到底有没有真实打到目标

## 7.4 结果观察层

目标：

- 只采集页面变化证据，不直接判成功失败

输入：

- 动作前页面状态
- 动作后页面状态

输出：

- `OutcomeEvidence`

建议字段：

1. `urlChanged`
2. `titleChanged`
3. `samePageUpdated`
4. `popupOpened`
5. `newTargetOpened`
6. `pageCountChanged`
7. `pageStateBefore`
8. `pageStateAfter`
9. `contentReadySignals`
10. `domMutationSummary`
11. `activePageId`
12. `activeTargetId`
13. `targetCreated`
14. `targetSwitched`

这层只回答：

- 页面后来到底发生了什么

## 7.5 结果判定层

目标：

- 根据动作意图与观察证据，产出最终结论

输入：

- `ActionAttempt`
- `OutcomeEvidence`
- 动作意图

输出：

- `ActionDecision`

建议状态：

1. `success`
2. `wrong_target`
3. `clicked_but_no_change`
4. `same_page_update`
5. `navigation`
6. `popup`
7. `new_target`
8. `blocked_by_overlay`
9. `blocked_by_verification`
10. `auth_required`
11. `target_not_bindable`

关键点：

- “动作已执行”与“页面已变化”必须是两个独立维度。

## 8. 关键数据结构建议

建议新增以下核心类型：

```ts
type ElementCandidate = {
  candidateId: string;
  semanticRole: string;
  text?: string;
  accessibleName?: string;
  title?: string;
  ariaLabel?: string;
  href?: string;
  name?: string;
  placeholder?: string;
  context: {
    containerSelector?: string;
    containerText?: string;
    rowIndex?: number;
    cardIndex?: number;
  };
  rect?: { x: number; y: number; width: number; height: number };
  riskTags: string[];
};

type BoundTarget = {
  targetId: string;
  runtimeNodeKey?: string;
  fingerprint: string;
  primaryAnchor: string;
  fallbackAnchors: string[];
  expectedRole?: string;
  expectedText?: string;
  expectedHref?: string;
  contextAnchor?: string;
};

type ActionAttempt = {
  executed: boolean;
  hitTarget: boolean;
  usedStrategy: string;
  clickedPoint?: { x: number; y: number };
  topElementAtPoint?: string;
  blockedBy?: string;
  note?: string;
};

type OutcomeEvidence = {
  urlChanged: boolean;
```

## 9. 第 1 轮到第 N 轮的状态流示例

## 9.1 第 1 轮：GitHub 文件列表点击

目标：

- 打开 `authors.yaml`

流程：

1. 页面发现层找出所有文件链接候选。
2. 目标绑定层把 `authors.yaml` 绑定为：
   - 文本：`authors.yaml`
   - `href`：目标文件页
   - 上下文：`#folder-row-9`
3. 动作执行层先做命中预检。
4. 若命中点上方元素不是目标，则直接返回：
   - `executed=false`
   - `blockedBy=other_element`
5. 若预检通过，才真正点击。
6. 结果观察层采集是否发生导航。
7. 结果判定层最终给出：
   - `success`
   - 或 `clicked_but_no_change`
   - 或 `wrong_target`

好处：

- 可以明确知道问题出在“绑定失败”“点击没落下去”还是“页面没变化”。

## 9.2 第 2 轮：Wikipedia 搜索

目标：

- 从首页进入词条页

流程：

1. 页面发现层识别输入框和提交控件。
2. 目标绑定层分别绑定：
   - 搜索输入框
   - 搜索按钮
3. 动作执行层先完成输入，再单独验证输入已写入。
4. 点击提交按钮后，动作执行层只回答“点中了没有”。
5. 结果观察层负责记录：
   - URL 是否跳转到词条页
   - 标题是否变化
   - 内容区是否就绪
6. 判定层才决定最终是否成功。

好处：

- 页面已经跳到词条页时，不会因为判定链太晚收敛而回头重试旧按钮。

## 9.3 第 3 轮：遮挡层关闭

目标：

- 关闭普通弹窗或遮挡层

流程：

1. 页面发现层识别弹窗根节点、关闭候选、空白区域。
2. 目标绑定层把“关闭按钮”或“空白关闭区域”绑定成真实目标。
3. 动作执行层先判断当前点击点会不会误打到 CTA。
4. 观察层只采证据：
   - 弹窗是否消失
   - 页面是否恢复可操作
5. 判定层输出：
   - `success`
   - 或 `blocked_by_overlay`
   - 或 `auth_required`

好处：

- 避免把“登录/继续”之类 CTA 当关闭按钮误点。

## 13. 边界与不变式

重构后应长期保持下面这些不变式：

1. 不允许直接拿模糊候选做最终动作目标。
2. 不允许没有 preflight 就直接点击或输入。
3. 不允许把“动作是否命中”和“页面是否变化”混成一个布尔值。
4. 不允许在页面已切走后继续拿旧目标盲目重试。
5. 不允许把 `overlay / verification / auth` 混成同类失败。
6. 不允许让上层 plan 继续直接消费脆弱 selector。

## 14. 为什么这条方案值得做

因为它直接对应了最近实测里暴露出的根问题：

1. GitHub 文件链接问题，本质是“元素身份”和“点击命中”没分开。
2. Wikipedia 提交问题，本质是“页面观察”和“成功判定”没分开。
3. 遮挡层问题，本质是“页面状态分类”和“真实动作目标”没分开。

继续补 heuristic，只会在不同站点上重复撞到同类问题。  
把这 5 层拆开后，失败至少能明确知道是哪一层错了，后续修复成本才会下降。

## 16. 当前结论

当前这套实现的核心问题，不是单个站点不好做，而是底层职责划分不清。

后续主线不应再是：

- 继续补 selector
- 继续补 heuristics
- 继续补站点特判

而应改成：

1. 先拆层
2. 再重建元素身份
3. 再重建动作执行
4. 最后重建观察与判定

这才是当前项目下一阶段最值得投入的方向。
