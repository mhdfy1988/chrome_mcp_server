## 2. 为什么不再作为主线

这份增量方案解决的是“现有架构还能不能再稳一点”，而不是“现有架构本身有没有根问题”。

它适合处理的问题是：

1. 增加日志
2. 修补资源释放
3. 优化超时与重试
4. 改善新页追踪

但最近几轮真实站点实测暴露的问题更底层：

1. 元素身份不稳定
2. 动作执行缺少命中证明
3. 页面观察与成功判定混在一起

这些问题不是继续补日志和补守卫就能彻底解决的，所以主线已经切到“拆层重构”。

## 3. 本文还保留什么价值

虽然不再是主线，但本文里仍有一些内容可以作为阶段间临时补丁参考：

1. 新页 / 新 target 跟踪增强
2. HTTP 传输资源关闭和清理
3. 调试日志增强
4. 超时与重试文档化

也就是说：

- 这些改动仍然可以做
- 但它们应被视为“过渡期稳态优化”，而不是“当前重构方案本身”

## 4. 建议如何使用本文

后续如果出现下面这类需求，可以回看本文：

1. 主架构还没改完，但需要先补日志便于排查
2. HTTP 模式或 stdio 模式存在资源释放问题
3. 某个慢站点需要临时拉长超时并保留更强观测

如果需求属于下面这类，则不应继续沿本文推进，而应直接回到主方案：

1. `ref` 为什么会串
2. 点击为什么没打到目标
3. 页面已经变了为什么还没判成功
4. 为什么 plan 的底座不稳

## 5. 当前结论

这份文档现在的角色是：

- 历史归档
- 增量补丁参考

它不是当前浏览器动作重构的并列方案，也不再参与主线决策。
