# chrome_mcp_server 项目规则

本文件补充当前仓库自己的浏览器自动化约束；未覆盖的通用规则，继续遵循上层 `AGENTS.md`。

## 浏览器动作验证

1. 对点击、提交、链接跳转这类浏览器动作做验证时，不能默认变化一定发生在当前 `page` 句柄上。
2. 动作后至少同时考虑这几类结果：
   - 当前页同页跳转或局部刷新
   - 弹出新页（`popup`）
   - 产生新 `target`
   - 活动页切换或页面数量变化
3. 如果用户肉眼已经看到跳转，而脚本仍显示“未变化”，优先检查是否出现了新页、新 `target` 或活动页切换，再判断动作是否失败。
4. 在没有排除新页、新 `target`、活动页切换之前，不得直接把浏览器动作判定为失败。
5. 浏览器工具返回结果时，应优先返回“真正发生变化的页面”，而不是机械地返回动作发起前的旧页句柄。

## 可见浏览器测试

1. 如果本轮测试依赖用户肉眼确认浏览器行为，不要在结果刚出现后立即自动关闭浏览器。
2. 需要关闭测试浏览器时，应先说明当前看到的页面结果，再关闭；如果用户正在观察，应优先等用户确认后再关闭。

## 入口与会话对齐

1. 如果用户明确要求基于“当前浏览器 / 当前会话 / 当前入口”继续操作，不得因为工具桥断开或调用失败，就擅自启动新的 Chrome 远程调试实例（例如 `--remote-debugging-port=9222`）、新的 `userDataDir` 或新的浏览器入口去替代当前入口。
2. 浏览器工具通道一旦断开，首先要明确说明“当前操作通道不可用”，而不是立刻切到 Puppeteer 脚本、自建 `browserURL`、临时调试端口或其他浏览器类型继续做所谓“同一轮实测”。
3. 如果确实需要临时验证入口，必须先重新对齐这 3 件事再执行：
   - 当前主入口是什么
   - 临时入口是什么
   - 是否会影响用户当前正在看的页面或会话
4. 在用户没有明确同意前，临时验证入口不能替代当前主入口，也不能被包装成“只是继续当前测试”。
5. 如果已经误开了临时入口或临时远程调试端口，必须立刻说明并优先清理掉，再回到主入口对齐，而不是沿着临时入口继续扩展操作。
6. “重启浏览器实例”和“重启 MCP stdio 进程”不是一回事：`close_browser` / `open_page` 只会重建浏览器实例，不等于重启 `stdio` 传输进程；如果用户明确要求“重启 MCP / 重启 stdio”，必须明确区分自己做的是哪一种。
7. 需要由助手主动启动项目自己的 `stdio` 入口时，使用 `npm.cmd run start:stdio`；执行后要再检查对应 `node dist/index.js --transport stdio` 进程是否仍然存活，不能只看到一行 `STDIO server started` 就当成已经稳定重启完成。

## 真实站点验证

1. 如果公开站点在 `headless` 模式下直接进入 `Cloudflare`、`Just a moment...` 或其他客户端挑战页，先归因为“入口验证层”问题，不要直接把失败归到输入框定位、提交计划或结果识别链路。
2. 这类场景需要交叉验证时，优先使用独立 `userDataDir` 的可见浏览器实例再试一次；验证目标是区分“站点防护”与“MCP 主链失败”，不是静默替换当前主入口。
3. 如果页面已经命中验证页或挑战页，应单独标记为 `blocked_by_verification`；若页面文案属于“Checking your browser / redirect shortly / please wait”这类自动放行提示，先等待几秒再判断是否需要人工介入，不要一上来就默认必须人工接管。
4. 如果等待后仍停留在验证页，再保留当前浏览器会话给人工接管；不要继续在这类页面上盲目做输入框定位、提交或结果验证。
5. 人工完成验证后，应继续复用同一个页面和会话往下执行，而不是默认重开浏览器或切换到新的临时入口。
6. 普通遮挡弹窗、Cookie 横幅、下载引导、活动弹窗这类页面遮挡，应与验证页分开处理；优先标记为 `overlay_blocking` 并先尝试自动关闭，不要直接归因到验证层。
7. 带明显登录门槛、登录墙、登录后查看更多、带密码输入框的登录拦截层，不应归到 `overlay_blocking`；应单独标记为 `auth_required`，不要继续尝试 `dismiss_blocking_overlays`。

## 构建与回归验证

1. 凡是依赖 `dist/` 产物的本地回归（如 `verify:plan-smoke`），必须先顺序执行 `npm.cmd run build`，再执行验证；不要把构建和验证并行跑。
2. 如果构建与验证并行执行，出现“代码看起来已改对，但 smoke 仍读到旧行为”的情况，优先怀疑验证读到了旧 `dist`，先改回顺序执行再判断功能是否失效。

## 遮挡层判定与关闭

1. 根节点若是 `pointer-events: none` 的穿透层，且内部没有真正的弹窗/表单/登录阻断子节点，不应判成 `overlay_blocking`；这类节点通常只是展示容器或视觉蒙层，不会真正拦截操作。
2. “右上角位置像按钮”不能单独当成关闭信号；只有明确关闭语义（如 `close/x/关闭/skip/later/cancel`）或强关闭图标信号成立时，才可以进入关闭候选。
3. 带 `登录/注册/login/register/sign in/continue` 语义的控件，默认视为高风险 CTA，不应作为 `dismiss_blocking_overlays` 的关闭候选。
4. 如果当前遮挡层没有明确关闭按钮，默认先尝试点击弹窗外空白区（`backdrop_click`），再尝试 `Escape`；不要先去点弹窗内部 CTA。
5. `backdrop_click` 的点击点要区分目标形态：目标是居中的 `modal` 时，应点击其外侧空白区；目标本身就是全屏遮罩时，才点击目标内部的安全边缘。

## 实战检验沉淀

1. 在真实站点上做页面理解、元素定位或动作验证时，必须优先以“当前页面事实”为准，不得直接沿用历史 selector 记忆，也不得凭标签名猜测元素语义。
2. 对搜索结果、视频列表、商品列表这类混合卡片页，优先点击“标题链接”这类强语义入口；不要脱离卡片上下文去点通用缩略图、泛化容器或重复出现的图标入口，以免误中广告、Shorts 或错误卡片。
3. 一旦某一层已经被证实成立，就先冻结这一层；后续失败只排查当前层，不要因为后面失败就回头改前面已经成立的输入框、链接或页面理解路径。
4. 搜索/查询提交流程要区分“输入框定位”和“提交触发”两层：输入框已确认后，如果结果没出来，先只切换提交策略（如 `Enter` / 点击明确提交按钮），不要重新怀疑输入框定位。
5. `preferredSubmitMethod=enter` 只能作为优先提示，不能当成站点真实行为的保证；真实站点若 `Enter` 未触发结果页，应改为点击明确提交按钮继续验证，而不是直接把问题归到页面理解层。
6. 对打开新页、弹窗或新 `target` 的站点，后续正文提取、字段抽取和摘要生成必须绑定到“真正发生变化的页面”；不能在旧页上继续提取并据此判断详情页失败。
7. 对商品页、文档页、帮助页这类详情页面，摘要优先走结构化字段或主内容区提取，不要默认从整页文本里硬抠；页头导航、相似推荐、广告和站点 chrome 应视为噪音。
8. 如果真实站点首页或结果页直接落到 `Cloudflare`、`Client Challenge`、`Just a moment...` 之类安全验证页，应明确归因为“入口验证层阻塞”，不要误记成 MCP 的搜索、点击或结果识别失败。
9. 表格行、单元格、列表项、卡片容器这类结构节点默认视为上下文容器，不应直接当成最终点击目标；执行前应优先收敛到其内部最强语义动作元素（如 `a[href]`、`button`、提交按钮），只有无法收敛时才保留容器本体。
